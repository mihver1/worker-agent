## Анализ проекта Worker

### 🟢 Сильные стороны

**1. Архитектура — продуманная и чистая**
Монорепо из 4 пакетов (`worker-ai`, `worker-core`, `worker-server`, `worker-tui`) с чёткими границами ответственности. Зависимости идут в одну сторону: `ai → core → server/tui`. Нет циклических импортов.

**2. Абстракция провайдеров — правильная**
`Provider` — чистый ABC с одним методом `stream_chat`. Добавить новый провайдер = написать один класс. `ProviderRegistry` — простой и рабочий.

**3. Агентный цикл — корректный**
`AgentSession._run_loop()` правильно обрабатывает: streaming, tool calls, steering (mid-run interrupts), follow-up очереди, abort через `asyncio.Event`. Это нетривиально и сделано аккуратно.

**4. Тесты — есть и работают**
239 тестов, все зелёные. Есть моки провайдеров, тесты WebSocket-протокола, REST API, OAuth, permissions, sessions. Это серьёзный плюс.

**5. Система расширений — элегантная**
`entry_points` + `@hook(event)` декоратор + `HookDispatcher` с pipeline-фильтрами. Горячая перезагрузка через `reload_extensions_async`. Чисто.

**6. Конфиг — удобный**
Pydantic-модели + TOML + deep merge (global → project). Шаблоны с комментариями на русском — хорошая DX.

**7. Persistence — нормальная**
SQLite через `aiosqlite`, схема с индексами, поддержка fork/branch сессий, компактинг через LLM.

---

### 🔴 Слабые стороны (честно)

**1. Permissions — система есть, но не подключена к агенту**
`PermissionPolicy` в `permissions.py` написана, но в `AgentSession` она **не используется**. `permission_callback` принимается в конструкторе, но нигде не вызывается в `_run_loop`. Конфиг `bash = "ask"` по умолчанию — но bash выполняется без спроса. Это дыра в безопасности.

**2. TUI — один гигантский файл (1129 строк)**
`app.py` делает всё: инициализацию сессии, все slash-команды, remote/local режимы, bash-выполнение, cmux-интеграцию. Это God Object. Любое изменение рискует сломать что-то несвязанное.

**3. Дублирование инициализации сессии**
Логика создания `AgentSession` (resolve config → resolve key → create provider → create tools → load extensions → create session) повторяется в трёх местах: `cli.py::_print_mode`, `app.py::_init_local_session`, `rpc.py::RpcServer._init_session`. Это нарушение DRY и источник расхождений.

**4. Сервер — TODO в продакшн-коде**

elif msg_type == "cancel":
    # TODO: implement cancellation
    pass
elif msg_type == "approve_tool":
    # TODO: implement permission approval
    pass

Клиент отправляет `cancel` — сервер молча игнорирует. Это не заглушка для будущего, это сломанная функциональность.

**5. aiohttp middleware — deprecated API**

DeprecationWarning: old-style middleware deprecated, see #2252

Тесты показывают 4 предупреждения. Middleware написан в старом стиле (`async def auth_middleware(app, handler)`), а не через `@web.middleware`. Не критично, но технический долг.

**6. Оценка токенов — грубая**
`_estimate_tokens()` считает `chars // 4`. Для кода это может ошибаться в 2-3 раза. Авто-компактинг может срабатывать слишком рано или слишком поздно. Нет привязки к реальному `context_window` модели при инициализации сессии.

**7. WebSocket в TUI — хрупкий**
`_run_remote` держит одно соединение `self._ws` на весь lifetime приложения. При разрыве соединения оно обнуляется (`self._ws = None`), но следующий запрос создаст новое соединение без восстановления состояния сессии на сервере. Сессия на сервере потеряна.

**8. `_resolve_api_key` в `server.py` — урезанная копия**
В `server.py` есть своя `_resolve_api_key` без OAuth-поддержки, которая импортирует `_ENV_KEY_MAP` из `cli.py` (приватный символ). Это плохая зависимость между пакетами.

**9. Нет rate limiting и backpressure на сервере**
`ServerState.sessions` — обычный dict, растёт бесконечно. `max_sessions` из конфига нигде не проверяется. Любой клиент может создать сколько угодно сессий.

**10. `_run_loop` — не настоящий `AsyncIterator`**
Метод объявлен как `async def _run_loop(self) -> AsyncIterator[AgentEvent]`, но внутри использует `yield`. Это работает, но тип в сигнатуре неточный — должен быть `AsyncGenerator`. Мелочь, но вводит в заблуждение.
